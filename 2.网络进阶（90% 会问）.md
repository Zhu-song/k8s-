# 一、Service4种类型（必背）
1.**ClusterIP**
- 默认类型
- 只在**集群内部**访问
- 外部不能访问
2.**Nodeport**
- 在每一个节点开一个端口
- **外部可以访问**
- 端口范围：30000–32767
3.**LoadBalancer**
- 云商提供负载均衡器
- 给一个**公网IP**
- 外部直接访问
4.ExternalName
- 把Service映射到外部域名
- 不再代理，不分配IP
# 二、kube-proxy是干什么的？
- 节点上的网络代理
- 实现**Service的负载均衡**
- 维护iptables或IPVS规则
## 两种模式（面试必问）
**1.iptables模式**
- 默认
- 性能一般，规则多了会卡
2.**IPVS模式**
- 性能更好
- 支持更多负载均衡算法
- 高并发集群用它
# 三、CNI是什么？
- **Container Network Interface**
- 容器网络标准接口
- 用来给Pod分配IP、组网、互通
### 常见的CNI组件
- **Flannel**：简单、稳定、适合入门
- **Calico**：性能好，支持NetworkPolicy网络策略
- **Weave**：方便，自带隧道
# 四、Pod跨主机通讯（一句话）
- 每个Node有自己的**PodCIDR网段**
- 不同节点的Pod通过**网关/隧道/路由**互相访问
- 所有Pod在一个**扁平网络**里，直接互通
# 五、Service怎么实现负载均衡？
1.Service有一个**固定的ClusterIP**
2.背后对应一组Pod（通过Label Selector）
3.kube-proxy在每个节点写**iptables/IPVS规则**
4.访问 Service IP → 转发到某一个 Pod
→ 实现**负载均衡**
# 六、NetworkPolicy网络策略
- 用来**限制Pod之间的流量**
- 允许/禁止：谁能访问我、我能访问谁
- 必须CNI支持（Calico支持，Flannel不支持）
