# 一、RBAC是什么
- **Role-Based Access Control**：基于角色的权限控制
- 作用：控制谁对哪些资源能进行什么操作
- 是K8s权限管理标准方案
# 二、RBAC四个核心资源（必背）
## **1.Role**
- 定义一组**权限**（如：查看Pod、创建Deployment）
- 只在**单个Namespace**内有效
## 2.**ClusterRole**
- 集群级别的角色
- 作用于**整个集群**，跨Namespace
## 3.**RoleBinding**
- 将**Role绑定到用户/ServiceAccount**
- 仅在当前Namespace有效
## 4.**ClusterRoleBinding**
 - 将**ClusterRole绑定到用户/ServiceAccount**
 - 整个集群有效
# 三、Subject 被授权的对象
 - User（用户）
 - Group（组）
 - **ServiceAccount（服务账号）**
	 - 给**Pod内部程序**使用的账号
	 - 用于容器内访问K8s API
# 四、RBAC三要素（一句话）
- **谁（Subject）**
- **能做什么（Role/ClusterRole）**
- **绑定关系（Binding）**
# 五、Secret安全问题（高频坑点）
- Secret只是**Base 64编码，不是加密**
- etcd中明文可查
- 生产必须：
	- 开启**etcd加密**
	- 配置**RBAC权限**限制访问
	- 使用外部密钥管理服务（如Vault）
# 六、ServiceAccount是什么
- 给**应用/Pod**用的“账号”
- 让容器内部有权限访问K8s API
- 默认每个Namespace都有`default`  ServiceAccount
# 七、NetworkPolicy网络策略
- 控制**Pod之间的流量**xu y
- 允许/禁止：谁能访问我、我能访问谁
- 需要CNICNI插件支持（Calico支持，Flannel不支持）
# 八、Pod安全基础（面试常问）
- 不使用Pod用户运行容器
- 不给容器unnecessary权限
- 启用**ResourceQuota**限制资源
- 启用**LimitRange**限制单Pod资源上限
