# 一、Deployment（无状态应用核心）
- **管理无状态服务**，最常用的高级资源
- 核心特点：
	- Pod名称随机（如xxx-7f89d-9x87z），无固定标识
	- Pod可随意替换、漂移，不绑定固定存储和网络
	- 扩缩容、更新、回滚均并行、无序列行
- 适用场景：Web服务、APi接口、网关、前端应用
# 二、StatefulSet（有状态应用核心）
- 管理**有状态服务**
- 核心特点：
	- Pod名称固定（如xxx-1，xxx-2，xxx-3），有序列号
	- 每个Pod有**固定网络标识**（固定DNS/hostname）
	- 每个Pod有**专属PV/PVC**，数据独立，不共享
	- 启动、删除、更新均为有序执行（从N-1到0，反向操作）
- 适用场景：数据库（MySQL、MongoDB）、集群类应用（Zookeeper、Elasticsearch）
# 三、DaemonSet（节点级服务）
- 核心作用：**每个节点必跑一个Pod**
- 核心特点：
	- Pod副本数由节点数决定，无法手动扩缩容
	- 新增加节点自动部署Pod，删除节点自动清理Pod
	- 可通过NodeSelector、污点容忍，指定仅在特定节点运行
- 适用场景：日志收集、节点监控、网络插件
# 四、Job（一次性任务）
- 核心：执行**一次性任务**，任务执行完完成后，Pod自动终止
- 核心特点：
	- 保证任务**至少执行一次**，执行失败后会自动重试
	- 任务完成后，Pod状态变为completed，不占用资源
- 适用场景：数据备份、数据导入、批量计算、应用初始化
# 五、CronJob（定时任务）
- 核心：基于**定时规则**，周期性触发Job执行
- 核心特点：
	- 使用Linux crontab（如*****，分别对应分、时、日、月、周）
	- 本质是“定时触发的Job”，继承Job的所有特性
	- 可配置并发策略，避免同一任务重复执行
- 适用场景：定时备份、定时清理日志、定时生成报表、定时巡检
